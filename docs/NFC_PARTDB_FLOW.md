# NFC 与 Part-DB 对象绑定流程

本文档描述当前固件中 NFC 卡片写入、读取和 UI 响应的约定。实现入口主要在 `firmware/main/device_ui.c`、`firmware/main/nfc_service.c` 和 `firmware/main/nfc_pn532.c`。

## 写入内容

固件在详情页写入 NFC 时，写入的是 NDEF Text Record，文本内容优先保存可直接解析的短路由。短路由用于适配当前容量很小的 NFC 卡，读取后会映射回 Part-DB 对象：

- `p/{id}`：Part-DB 元件路径别名，例如 `p/1`
- `l/{id}`：Part-DB 批次路径别名，例如 `l/8`
- `IPN`：当详情对象没有有效 id 时的兜底

选择规则：

- 当前详情对象是批次且有有效批次 ID 时，优先写入 `l/{id}`。
- 否则写入当前元件的 `p/{id}`。
- 如果没有有效 id，但有 IPN，则写入 IPN。
- 当前详情页没有查到有效对象时，不允许写入。

## 详情页写入流程

1. 用户进入元件或批次详情页。
2. 固件根据详情对象生成 `p/{id}`、`l/{id}` 或 IPN。
3. 用户点击详情页的 `写NFC`。
4. 屏幕弹出确认框，显示即将写入的 payload，并提示先放卡再确认；弹窗期间后台 NFC 轮询继续运行。
5. 用户确认后，UI 认领后台已经检测到的当前卡片，然后短暂停止后台轮询。
6. 固件调用 `nfc_pn532_write_ndef_text_to_tag(payload, 3000)` 写入路由或 IPN。
7. 写入完成后读回 NDEF 文本校验，随后恢复后台 NFC 轮询。
8. UI 显示 `已写入 p/{id}`、`已写入 l/{id}`、`已写入 IPN` 或具体错误码。

写入期间如果相机或其它请求占用外设，可能返回资源忙或超时。此时重新贴卡再点确认即可重试。

## 设置页擦除流程

设置页提供 `擦除 NFC` 功能，用于把卡片恢复为不绑定 Part-DB 对象的状态：

1. 用户进入设置页，点击 `擦除 NFC`。
2. 屏幕弹出确认框，提示会清空卡片 NDEF 内容。
3. 弹窗期间后台 NFC 轮询继续运行，用户确认后 UI 认领当前卡片并短暂停止后台轮询。
4. 固件写入空 NDEF TLV。
5. 擦除完成后恢复后台 NFC 轮询。

擦除后的卡片即使还能读到 UID，也不会触发详情页跳转，因为没有有效的 Part-DB 路由、IPN 或内部对象码文本。

## 读取和页面响应

后台 `nfc_service` 常驻轮询 PN532：

1. 读到卡片 UID 后，尝试读取 NDEF Text Record。
2. 读到文本后保存到 `nfc_service_status_t.text`。
3. UI 发现 `read_count` 增加且文本非空时，自动进入详情页。
4. 详情页将文本交给 Part-DB 解析链路：
   - `p/{id}` 查询 `/api/parts/{id}.jsonld`
   - `l/{id}` 查询 `/api/part_lots/{id}.jsonld`
   - 也兼容 `P####`、`L####` 和扫码桥返回的内部对象码

如果只读到 UID，没有读到 NDEF 文本，快捷页只显示 UID，不会自动跳详情。
空卡、已擦除卡或非文本 NDEF 卡只作为无效业务卡处理，不会把 PN532 标记为离线。

## 在线状态策略

UI 的 NFC 在线状态以后台服务 ready 或 PN532 驱动 ready 任一为准。后台服务在空闲健康检查中不会因为一次 ping 失败就立刻标记离线，连续多次失败才重新初始化 PN532，避免屏幕上 NFC 状态频繁跳离线。

## 实机验证清单

- 进入某个元件详情页，写入空白 NFC 卡，确认屏幕显示 `已写入 p/{id}`。
- 将卡片移开再贴回，确认快捷页显示已读卡，并自动跳转到对应详情页。
- 在设置页执行 `擦除 NFC`，确认再次贴卡只显示 UID 或空内容，不自动跳详情。
- 对有批次的对象写入，确认 payload 为 `l/{id}`，读取后进入批次详情。
- 在相机预览退出后贴卡，确认 NFC 状态恢复在线。
- 连续静置 2 分钟，确认 NFC 状态不会无故闪成离线。
